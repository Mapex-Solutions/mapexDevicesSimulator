# 13 — Internationalization (EN / PT-BR)

The whole interface is bilingual: every user-facing string is available in **English**
and **Brazilian Portuguese**, switchable at runtime.

## Capabilities

- **Two locales** — `en-US` and `pt-BR`, fully translated.
- **Runtime switch** — change the language from the app header, no reload needed.
- **Complete coverage** — there are no hardcoded user-facing strings; labels,
  placeholders, statuses and messages all go through translations.

## How it works

A translations composable (`useTranslations` → `t(key)`) reads the locale dictionaries
under `app/src/i18n/en-US` and `app/src/i18n/pt-BR`, which mirror each other key for
key. Components reference keys, never literals; a header control toggles the active
locale.

The API contracts and data model are language-agnostic — translation is purely a
presentation concern in the frontend.

## Notes

- The two dictionaries are kept structurally in sync, so a missing translation is easy
  to spot.
- Protocol-level tokens that are not prose (e.g. `qos1`, `EU868`, a DevAddr) are shown
  verbatim, not translated.
